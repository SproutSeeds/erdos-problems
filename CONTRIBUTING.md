# Contributing to erdos-problems

`erdos-problems` uses this GitHub repo as the central public collaboration space for:

- canonical dossier updates
- pack-specific route, board, and compute packets
- CLI and runtime changes
- documentation and workflow clarity improvements

## What lives where

- `problems/<id>/`: canonical local dossier truth for seeded problems
- `packs/<family>/`: pack-specific public packets and deeper harness context
- `src/`: CLI and runtime behavior
- `data/upstream/erdosproblems/`: bundled external import snapshot shipped with the package
- `.erdos/`: local workspace runtime generated by the CLI; useful for working sessions, not canonical repo truth

## Repo vs npm package

The npm package is the installable distribution of the CLI and packaged assets.

This GitHub repo is broader than the npm tarball:

- repo-only contribution guides and templates can live here without shipping in npm
- maintainers can keep collaboration structure in the repo even when it is not needed for local installs
- canonical public contribution happens through issues, discussions, and pull requests here

## Good first contribution paths

- improve a dossier under `problems/<id>/`
- add or tighten pack context under `packs/`
- improve CLI clarity in `README.md` or `src/commands/`
- review external import drift with `erdos upstream diff`
- seed a new problem from a pulled bundle with the maintainer flow

## Truth hygiene

Please preserve the package's truth split:

- canonical public truth for this project lives in `SproutSeeds/erdos-problems`
- imported public source material may come from `teorth/erdosproblems` and `erdosproblems.com`
- local canonical dossier truth lives in `problems/<id>/`
- local workspace truth lives under `.erdos/`
- pack packets guide the harness but should not silently upgrade global problem status

When unsure, prefer smaller, explicit claims:

- keep open problem, active route, route breakthrough, and problem solved distinct
- checkpoint load-bearing progress instead of widening claims early
- preserve failed paths as artifacts when they matter

## Useful commands

```bash
npm install
npm test
node bin/erdos upstream show
node bin/erdos upstream diff
node bin/erdos bootstrap problem 857
node bin/erdos sunflower frontier 857
```

To refresh the workspace-local external import snapshot:

```bash
node bin/erdos upstream sync
```

To intentionally refresh the bundled import snapshot in the repo as a maintainer:

```bash
node bin/erdos upstream sync --write-package-snapshot
```

## Pull request expectations

- explain which truth layer your change touches
- note whether you changed dossier data, pack data, or CLI/runtime behavior
- include command output or tests when you change CLI behavior
- call out any upstream drift or site-status implications explicitly
