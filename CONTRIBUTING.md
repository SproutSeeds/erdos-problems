# Contributing to erdos-problems

Thank you for building here.

This repo is the canonical public home for `erdos-problems`. It is where we keep the shared research record, the CLI, the pack structure, the paper bundles, and the contribution workflow. We want it to be rigorous, welcoming, and easy to work in.

This guide is our v1 public contribution contract. It is meant to be friendly, explicit, and predictable. If the process feels unclear, please open a clarity issue and we will tighten it.

## What this repo is for

Use this repo for public, research-relevant work such as:

- canonical dossier updates under `problems/<id>/`
- pack packets under `packs/<family>/`
- CLI and runtime changes under `src/`
- public formalization and paper bundles under `formal/` and `paper/`
- deeper public research notes under `research/` and `analysis/`
- contribution, review, and workflow improvements

Do not use this repo for:

- secrets, credentials, or private access details
- unpublished private correspondence
- local machine paths or personal environment state
- overclaimed status updates that are not supported by the current public record

## Core principles

### 1. Keep the truth split clean

For this project:

- canonical repo truth lives in `SproutSeeds/erdos-problems`
- external import sources may inform the repo, but they do not outrank it
- workspace state under `.erdos/` is local runtime state, not canonical public truth
- pack progress does not silently upgrade a problem from open to solved

### 2. Smaller honest claims beat larger vague claims

When in doubt:

- downgrade the claim level
- show the exact evidence
- name the open seam
- preserve failed attempts as useful artifacts

### 3. Review is part of the contribution

Contributions here are gated on review. That is not a punishment. It is how we keep the public research record trustworthy and reusable for the next collaborator.

### 4. Templates are required

If you want to share evidence, a patch, a correction, or a proposal, please use the relevant issue or PR template. If a submission skips the template or leaves out critical fields, we may reply with a reminder, close the submission, and ask for a resubmission in the proper format. Once the right template is used, we can reopen or continue from there.

## Start here

Before opening anything:

1. Read the relevant local files first.
2. Search existing issues and pull requests.
3. Decide which truth layer your change touches.
4. Choose the right template.
5. Make the smallest honest submission that moves the work forward.

Useful commands:

```bash
npm install
npm test
node bin/erdos import show
node bin/erdos import diff
node bin/erdos problem show 857
node bin/erdos problem artifacts 857 --json
node bin/erdos paper show 857
```

To refresh the workspace-local external import snapshot:

```bash
node bin/erdos import sync
```

To intentionally refresh the bundled import snapshot in the repo as a maintainer:

```bash
node bin/erdos import sync --write-package-snapshot
```

## What lives where

- `problems/<id>/`: canonical dossier truth for seeded problems
- `packs/<family>/`: pack-specific route, board, ticket, atom, and compute packets
- `src/`: CLI and runtime behavior
- `data/upstream/erdosproblems/`: bundled external import snapshot for provenance and seeding
- `research/`: repo-only deep public research bundles
- `formal/`: repo-only formalization work intended to be public
- `paper/`: repo-only paper bundles and publication scaffolding
- `analysis/`: repo-only public analysis artifacts
- `.erdos/`: local runtime state generated by the CLI; useful for working sessions, not canonical repo truth

## Open an issue or open a PR?

### Open an issue first when

Open an issue if you are:

- sharing new evidence
- proposing a dossier change
- proposing a pack or route update
- reporting drift between the repo and an external source
- proposing a new compute, formalization, or paper artifact
- making a nontrivial CLI or workflow change
- unsure whether a change should land at all

### Open a pull request when

Open a pull request when you already have a concrete patch.

For this repo, the expected default is:

- work on a branch in your fork or in the main repo if you have access
- open the pull request as a draft first
- keep it draft until the template is filled out, the scope is clear, and checks are green
- mark it ready only when it is ready for review, not when you want feedback on a half-described change

Small doc typos or obvious wording fixes may go straight to a draft PR without a prior issue. Everything else should usually have an issue first.

## Branch and draft PR workflow

If you are contributing code, docs, dossiers, packs, or paper material:

1. Create a branch for the work.
2. Make the change on that branch, not on `main`.
3. Open a draft PR.
4. Fill in the entire PR template.
5. Run the relevant commands locally.
6. Wait for CI and automated review to come back clean.
7. Move the PR from draft to ready when the submission is genuinely reviewable.

If you are not sure whether a change is big enough for a draft PR, err on the side of opening one. A good draft PR is easier to review than a large surprise.

## How to present evidence

If you are sharing mathematical evidence, route evidence, formal evidence, or computational evidence, please make it legible and reviewable.

At minimum, include:

- the problem ID
- the current route or pack context if relevant
- the exact claim you want the repo to take seriously
- the claim level: `Exact`, `Verified`, `Heuristic`, or `Conjecture`
- the canonical files or packets you think are relevant
- the source surfaces you reviewed
- the verification hook or reproduction path
- the next honest move if the evidence is incomplete

If useful, follow the ORP bundle language directly:

- [templates/CLAIM.md](/Volumes/Code_2TB/code/erdos-problems/templates/CLAIM.md)
- [templates/VERIFICATION_RECORD.md](/Volumes/Code_2TB/code/erdos-problems/templates/VERIFICATION_RECORD.md)
- [templates/FAILED_TOPIC.md](/Volumes/Code_2TB/code/erdos-problems/templates/FAILED_TOPIC.md)

Good evidence submissions are specific. They do not say “I think this solves it.” They say exactly what changed, exactly what is supported, and exactly what remains open.

## Evidence expectations by kind

### Dossier evidence

For `problems/<id>/` changes, say:

- what fact is being added or corrected
- what public source supports it
- whether the change affects status, formalization, references, or statement wording

### Pack evidence

For `packs/` changes, say:

- which route, ticket, atom, or compute lane is affected
- whether the change is operational, evidentiary, or status-relevant
- whether it changes the next honest move

### CLI or docs evidence

For `src/`, `README.md`, or docs changes, say:

- what was confusing or broken
- how the user experiences it now
- what command or page shows the improvement

### Paper or formal evidence

For `paper/` or `formal/` changes, say:

- whether the change is exposition, structure, formalization, verification, or bibliography
- whether the section uses only public materials
- whether the writing is still claim-safe

## Review flow

This is the usual path once an issue or draft PR is opened.

### 1. Intake check

We check that:

- the right template was used
- the scope is understandable
- the truth layer is named clearly
- the submission is public-safe

If not, we may leave a reminder and close the item until it is resubmitted in shape.

### 2. Evidence check

We check that:

- the claim level is honest
- the cited evidence matches the claim
- external sources are treated as imported provenance, not automatic authority
- the submission does not silently widen problem status

### 3. Structure check

We check that:

- files are in the right place
- dossier, pack, paper, and formal boundaries stay clean
- new artifacts fit the style of the repo

### 4. Automation and CI check

We check that:

- the PR is still in draft until it is ready
- CI is green
- required local commands are listed
- bots or automated review notes have been addressed

### 5. Maintainer review

We check for:

- truth hygiene
- evidence quality
- compatibility with the current repo structure
- whether the change should merge now or be split into smaller pieces

### 6. Merge or next step

Once the review gates are satisfied, we either:

- merge the change
- ask for a revision
- split the work into a follow-up issue
- archive the attempt as useful but not yet mergeable

## What happens if a submission misses the template?

Usually:

1. we leave a reminder pointing to the correct template or missing sections
2. if the submission still is not usable, we may close it
3. once it is resubmitted in the right format, we continue review

This is a gated interaction model. The goal is not to block contributors. The goal is to make the public record clean enough that people and agents can pick up where someone else left off without guessing.

## Pull request checklist before marking ready

Before changing a draft PR to “ready for review,” please make sure:

- the PR links the relevant issue when there is one
- the summary is specific
- the truth layer touched is named clearly
- evidence and claim level are stated honestly
- relevant commands or tests were run
- CI is green
- no secrets, local machine paths, or private correspondence are present
- the branch is up to date enough for review

## Etiquette

Please be generous and direct.

- be kind to other contributors
- prefer precise criticism over vague rejection
- quote files, commands, and sources exactly
- ask for the smallest next change that would make a contribution reviewable
- treat failed attempts as useful research artifacts when they teach us something

## Good first contributions

- improve a dossier under `problems/<id>/`
- add or tighten pack context under `packs/`
- improve CLI clarity in `README.md` or `src/commands/`
- review external import drift with `erdos import diff`
- seed a new problem from a pulled bundle with the maintainer flow
- improve a paper bundle under `paper/problems/<id>/`
